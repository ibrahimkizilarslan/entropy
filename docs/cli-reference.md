# CLI Reference

Entropy is managed entirely through its Command Line Interface (CLI). Below is a comprehensive list of all commands and their functionalities.

## General Commands

### `entropy init`
Scans the current directory for a `docker-compose.yml` file and automatically generates a default `chaos.yaml` configuration file populated with the discovered services as targets.

*   **Usage:** `entropy init`
*   **Options:** None.

### `entropy start`
Starts the background chaos engine using the random injection model defined in `chaos.yaml`. The engine will periodically inject faults based on the configured intervals and safety constraints.

*   **Usage:** `entropy start [OPTIONS]`
*   **Options:**
    *   `--config <path>`: Path to a custom configuration file (default: `chaos.yaml`).

### `entropy stop`
Stops the running background chaos engine. This will gracefully terminate the daemon process but it will *not* automatically revert active faults (use `entropy cleanup` for that).

*   **Usage:** `entropy stop`

### `entropy status`
Displays the current status of the background chaos engine, including whether it is running, its PID, and the target configuration it was started with.

*   **Usage:** `entropy status`

### `entropy logs`
Streams the logs generated by the background chaos engine in real-time. Useful for monitoring which faults are being injected and observing safety constraint violations.

*   **Usage:** `entropy logs [OPTIONS]`
*   **Options:**
    *   `--lines <number>`: Number of recent log lines to display initially.

### `entropy cleanup`
Emergency cleanup command. Reverts all active faults that Entropy might have injected. It removes network constraints (`tc` rules) and restores resource limits (CPU/Memory) across all targeted containers.

*   **Usage:** `entropy cleanup`

## Scenario Commands

The Scenario Engine allows for deterministic, hypothesis-driven testing rather than random chaos.

### `entropy scenario run <file>`
Executes a deterministic chaos scenario defined in a YAML file. It runs the steps sequentially, evaluates HTTP probes, and reports whether the scenario hypothesis passed or failed.

*   **Usage:** `entropy scenario run <path/to/scenario.yaml>`
*   **Arguments:**
    *   `<file>`: The path to the scenario YAML file (e.g., `chaos-scenario.example.yaml`).

## Docker Inspection (Optional/Debug)

### `entropy docker list`
Lists all running Docker containers on the host machine. This is a utility command to verify Docker connectivity and see what containers are available to target.

*   **Usage:** `entropy docker list`
