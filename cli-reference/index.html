    CLI Reference

    Entropy is managed entirely through its Command Line Interface (CLI). Below is a comprehensive list of all commands and their functionalities.

    General Commands

    entropy init

    Scans the current environment (Docker Compose or Kubernetes) and automatically generates a default chaos.yaml configuration file populated with the discovered services/deployments as targets.

    • Usage: entropy init
    • Options:
      • --runtime <docker|kubernetes>: Container runtime to use (default: docker).
      • --force, -f: Overwrite chaos.yaml if it exists.

    entropy start

    Starts the background chaos engine using the random injection model defined in chaos.yaml. The engine will periodically inject faults based on the configured intervals and safety constraints.

    • Usage: entropy start [OPTIONS]
    • Options:
      • --config <path>: Path to a custom configuration file (default: chaos.yaml).
      • --runtime <docker|kubernetes>: Container runtime to use (default: docker).
      • --detach, -d: Run engine in background mode.
      • --dry-run: Override config and log actions without executing them.
      • --max-down <number>: Override safety.max_down.
      • --cooldown <seconds>: Override safety.cooldown.

    entropy stop

    Stops the running background chaos engine. This will gracefully terminate the daemon process but it will not automatically revert active faults (use entropy cleanup for that).

    • Usage: entropy stop

    entropy status

    Displays the current status of the background chaos engine, including whether it is running, its PID, and the target configuration it was started with.

    • Usage: entropy status

    entropy logs

    Streams the logs generated by the background chaos engine in real-time. Useful for monitoring which faults are being injected and observing safety constraint violations.

    • Usage: entropy logs

    entropy cleanup

    Emergency cleanup command. Reverts all active faults that Entropy might have injected. It removes network constraints (tc rules) and restores resource limits (CPU/Memory) across all targeted containers.

    • Usage: entropy cleanup

    entropy inject [action] [target]

    Manually injects a single chaos action into a target container.

    • Usage: entropy inject [action] [target] [OPTIONS]
    • Arguments:
      • [action]: One of stop, restart, pause, delay, loss, limit_cpu, limit_memory.
      • [target]: Docker container/service name or Kubernetes deployment/pod name.
    • Options:
      • --runtime <docker|kubernetes>: Container runtime to use (default: docker).
      • --config, -c <path>: Config file path used for target allow-list checks.
      • --skip-validation: Skip target allow-list check.
      • --latency <ms>: Delay latency (for delay).
      • --jitter <ms>: Delay jitter (for delay).
      • --loss <percent>: Packet loss percentage (for loss).
      • --cpu <cores>: CPU quota value (for limit_cpu).
      • --memory <mb>: Memory limit in MB (for limit_memory).
      • --duration <seconds>: Auto-restore duration.

    entropy doctor

    Analyzes local Docker Compose or Kubernetes topology for resilience risks. It specifically checks for: - SPOF: Services with less than 2 replicas. - Resource Limits: Missing CPU or Memory limits. - Recovery: Missing restart policies (Docker). - Observability: Missing Liveness/Readiness probes (Kubernetes) or Healthchecks (Docker). - Security: Containers running in privileged mode.

    • Usage: entropy doctor
    • Options:
      • --runtime <docker|kubernetes>: Container runtime to use (default: docker).

    entropy topology

    Visualizes discovered topology and network blast radius.

    • Usage: entropy topology

    entropy completion

    Generates shell completion scripts.

    • Usage: entropy completion [bash|zsh|fish|powershell]

    Scenario Commands

    The Scenario Engine allows for deterministic, hypothesis-driven testing rather than random chaos.

    entropy scenario run <file>

    Executes a deterministic chaos scenario defined in a YAML file. It runs the steps sequentially, evaluates HTTP probes, and reports whether the scenario hypothesis passed or failed.

    • Usage: entropy scenario run <path/to/scenario.yaml>
    • Arguments:
      • <file>: The path to the scenario YAML file (e.g., chaos-scenario.example.yaml).
    • Options:
      • --runtime <docker|kubernetes>: Container runtime to use (default: docker).
      • --report <html|json>: Report output format (default: html).
      • --output <path>: Write scenario report to disk.

    Docker Inspection (Optional/Debug)

    entropy docker list

    Lists all running Docker containers on the host machine. This is a utility command to verify Docker connectivity and see what containers are available to target.

    • Usage: entropy docker list

    Environment Variables

    Variable Description Default
    ENTROPY_K8S_NAMESPACE Target namespace for Kubernetes operations. default
    ENTROPY_NET_INTERFACE The network interface name inside containers where tc rules are applied. eth0
    KUBECONFIG Path to the kubeconfig file for cluster access. ~/.kube/config
